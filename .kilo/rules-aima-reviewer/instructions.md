# AIMA Reviewer 规则

## 首次使用说明

**重要**: 首次切换到此模式时，请手动选择模型 `glm-5`
Kilo 会记住此选择（Sticky Models 机制）

## 审查流程

### 1. 获取待审查文件
从任务描述或上下文中确定待审查的文件列表

### 2. 阅读相关文档
- `AGENTS.md` - 开发规范
- `docs/ARCHITECTURE.md` - 架构设计
- `docs/reference/domain/{domain}.md` - 领域规范

### 3. 执行审查
按审查清单逐项检查

### 4. 输出审查报告

## 审查清单

### 架构符合性
- [ ] 符合 Command/Query/Event/Resource 接口
- [ ] 命名符合 `{domain}.{action}` 规范
- [ ] 文件位置符合目录结构

### 代码质量
- [ ] 错误处理完整（不吞错误）
- [ ] 有上下文信息
- [ ] 命名清晰一致
- [ ] 无重复代码

### 安全性
- [ ] 输入验证
- [ ] 无敏感信息硬编码
- [ ] 无注入风险

### 测试
- [ ] 覆盖率达标
- [ ] 测试有意义

## 审查结果

### 通过
```markdown
## 审查结果: ✅ 通过

代码质量良好，符合项目规范。

### 小建议（可选）
- [非阻塞的建议]
```

### 需修改
```markdown
## 审查结果: ⚠️ 需修改

### 必须修复
1. [问题] (高)
   - 位置: file.go:123
   - 建议: [修复方法]

### 建议改进
- [非阻塞建议]
```

### 退回
```markdown
## 审查结果: ❌ 退回

### 重大问题
1. [严重问题]

需要重新设计方案后实现。
```
